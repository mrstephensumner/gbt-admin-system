# Implementation Plan: Profile Enrichment — per-site AI biographies

**Branch**: `013-profile-enrichment` | **Date**: 2026-07-23 | **Spec**: [spec.md](spec.md)

## Summary

Per-(talent × network-site) AI biography, generated by the Anthropic API from the talent's real
facts + the site's editorial brief, in British English, avoiding a banned-words list, reviewed and
dual-approved (admin + talent) before publishing. The org Anthropic key is stored AES-GCM-encrypted
(Worker KEK secret), never returned. A dependency-free trigram-Jaccard similarity score and
banned-word scan are surfaced as reviewer flags. Published bios are the system's first publish-safe
content; drafts/settings/key are internal-only. Migration 0011. New ADR 0007 (secret storage +
LLM-integration boundary).

## Technical Context

**Language/Version**: TypeScript, React 19, Hono on Cloudflare Workers.
**Primary Dependencies**: Hono, Drizzle/D1, WebCrypto (AES-GCM), `fetch` to the Anthropic Messages
API, React Router, design-system, Lucide.
**Storage**: D1 — `enrichment_settings` (single row), `talent_site_bio`, brief columns on `brand`,
`source_material` column on `talent` (migration 0011). Secret: `ENRICHMENT_KEK` (Worker secret).
**Testing**: Vitest (unit — similarity, banned-word scan, word count, prompt builder, state machine,
crypto round-trip), vitest-pool-workers (integration — settings key-never-leaks, generation via
`fetchMock` stubbing api.anthropic.com, approval gate, publish-safe guard, change records),
Playwright (e2e — settings, generate (server stubbed), approve, publish).
**Target Platform**: Single Worker (SPA + API) behind Access.
**Project Type**: Web application, established `app/` layout.
**Performance Goals**: Generation is one outbound API call per bio, on demand; similarity/word-count
are pure local functions.
**Constraints**: Key encrypted at rest, never returned/logged/public; generation grounded (no
invented facts); only published bios publish-safe; British English enforced by prompt.
**Scale/Scope**: Per-talent × per-site; one settings row; brief per site; one tab + one owner-only
settings screen; ~6 endpoints.

```text
│   ├── services/serialize.ts            # publish-safe read helper for published site bios (never key/drafts)
│   └── routes/enrichment.ts             # NEW — settings (owner), brief (admin), generate/edit/approve/publish, list
├── src/
│   ├── routes/enrichment-settings.tsx   # NEW — owner-only AI settings screen
│   ├── routes/enrichment-tab.tsx        # NEW — per-site enrichment cards + generate/review/approve/publish
│   ├── routes/network.tsx               # + editorial-brief fields in the site dialog
│   ├── routes/talent-profile.tsx        # wire real Profile Enrichment tab; +history labels
│   ├── routes/root.tsx                  # +owner-only nav to AI settings
│   └── lib/types.ts                     # +enrichment types
└── tests/{unit,integration,e2e}/…       # enrichment.* across the three tiers
```

**Structure Decision**: Extend the `app/` web app. The only genuinely new surfaces are (a) an
outbound LLM call isolated in `worker/lib/anthropic.ts` (one function, stubbed in tests) and (b)
AES-GCM crypto in `worker/lib/crypto.ts` — both small and unit-tested. Everything else follows the
established shared-vocab / service / route / tab / three-tier-test shape.
